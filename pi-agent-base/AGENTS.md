## Executing Skills and Commands

**When the user invokes a skill or slash command, execute it immediately.** The act of invoking a skill IS the instruction to run it — do not ask the user if they want you to run it, acknowledge that you received it, or summarize what it does before acting. Start Step 1 of the skill right away.

If the skill requires a target or argument that was not provided (e.g. `/debug` with no bug description, `/review` with no file), ask for it once using `ask_user_question` and then proceed immediately.

## Asking the User Questions

**Always use the `ask_user_question` tool** when you need to ask the user anything structured — choices between options, clarifying questions, or decisions before acting.

Never write question blocks, numbered question lists, or multi-part questions as plain text output. Route them through `ask_user_question` instead.

**When to use it:**
- Any choice between two or more options
- Confirming decisions before taking action
- Clarifying scope, preferences, or intent
- Gathering requirements

**When plain text is fine:**
- A single yes/no follow-up mid-conversation
- Asking the user to paste output or a value

**Schema** — always pass a `questions` array (1–4 questions). Each question:
```json
{
  "question": "Which approach should we use?",
  "header": "Approach",
  "options": [
    { "label": "Option A", "description": "What this means and its trade-offs" },
    { "label": "Option B", "description": "What this means and its trade-offs" }
  ],
  "multiSelect": false
}
```

Rules:
- `header`: short chip label, max 12 chars (e.g. "Approach", "Auth", "Scope")
- `options`: 2–4 choices. **Always include a `description`** — it explains what the option means in context. Do NOT add an "Other" option — that is provided automatically.
- `multiSelect: true` when choices are not mutually exclusive
- If you recommend an option, list it first and add "(Recommended)" to its label
- **Never call this tool multiple times in sequence** — bundle all questions into one call

## Pi Extension Compatibility

**Editor Component Conflicts**: Some Pi extensions register custom editors via `ctx.ui.setEditorComponent()`. Only ONE can be active — whichever loads last wins.

**Known conflicts:**
- `pi-vim` (ModalEditor) vs `pi-powerline-footer` (BashModeEditor)

**Solution**: Control package load order in `settings.json`. List the preferred editor LAST:

```json
{
  "packages": [
    "npm:pi-powerline-footer",
    "npm:pi-vim"  // ← Loads last, wins editor component
  ]
}
```

**Current config**: `pi-vim` loads last → Vim keybindings active + Powerline UI elements (footer/header/widgets) remain functional.

**See**: `~/.pi/agent/extensions/README-VIM-POWERLINE.md` for detailed explanation.
