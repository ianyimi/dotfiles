## brew-install Prompt Created — 18:45

Created custom prompt `/brew-install` to automate package research and Ansible playbook updates.

**Purpose:**
- Takes package name(s) as input
- Researches installation method (Homebrew cask/formula, npm, curl, custom)
- Determines if it's a GUI app or CLI tool
- Adds appropriate Ansible task(s) to `dot_bootstrap/macos.yml`
- Handles custom taps, version pinning, multi-step installs
- Skips duplicates silently

**Classification logic:**
- GUI applications → "Brew Install (Apps)" section
- CLI tools/libraries → "Brew Install (CLI)" section
- Based on *what* the package is, not *how* it's installed

**Research strategy:**
1. Try `brew search` and `brew info` first (fastest)
2. If not in Homebrew, research online (GitHub README, official docs)
3. Determine correct installation method and section
4. Generate appropriate Ansible task syntax

**Examples covered:**
- Simple Homebrew cask (raycast)
- CLI tool with custom tap (aerospace via nikitabobko/tap)
- npm package (typescript)
- Direct curl install (bun)

This replaces the need for the suggested `add-package` prompt — `brew-install` covers all package types.

## Pi Extension Conflict Debugging — 19:45

**Problem discovered:** `pi-vim` and `pi-powerline-footer` were conflicting. Only one would work at a time - toggling powerline on/off would switch which was active.

**Root cause:** Both extensions call `ctx.ui.setEditorComponent()` to register their custom editors:
- `pi-vim`: Registers `ModalEditor` (vim keybindings) 
- `pi-powerline-footer`: Registers `BashModeEditor` (bash mode + ghost suggestions)

Only ONE editor component can be active. Whichever loads last wins.

**Solution:** Control package load order in `~/.pi/agent/settings.json` so `pi-vim` loads LAST:

```json
{
  "packages": [
    "npm:pi-powerline-footer",
    "npm:pi-btw",
    "npm:pi-docparser", 
    "npm:pi-image-preview",
    "npm:lsp-pi",
    "npm:pi-vim"  // LAST → wins editor component
  ]
}
```

**Result:**
- ✅ Vim modal editing works (hjkl, dd, yy, etc.)
- ✅ Powerline status bar shows at top
- ✅ Vim mode indicator shows at bottom right
- ❌ Lost: Powerline's BashModeEditor features (bash mode, ghost suggestions)

**Trade-off accepted:** Vim keybindings + powerline UI is more valuable than bash mode features.

**Ansible playbook updated:**
- Reordered `pi install` commands so `pi-vim` installs last
- Added `quietStartup: true` configuration to show powerline footer immediately on startup
- Documented in `~/.pi/agent/extensions/README-VIM-POWERLINE.md`

**Future improvement:** Submit PR to `pi-powerline-footer` to make `BashModeEditor` extend `ModalEditor` when `pi-vim` is detected, allowing both to work together natively.

  // pi-vim not installed, use CustomEditor

  // Now inherits vim modal editing when available!
}

## Agent Documentation Update - Pi Workflow Clarification — 20:15

Updated agent context to clearly document the pi-agent-base/ workflow.

**Problem:** Agents were editing `~/.pi/agent/` files directly, which get overwritten on `chezmoi apply`.

**Solution:** Updated `.pi/AGENTS.md` and `.pi/agent-docs/product/tech-stack.md` with clear workflow rules:

**Key points:**
1. **Source of truth:** `pi-agent-base/` (in chezmoi source)
2. **Synced target:** `~/.pi/agent/` (on local machine)
3. **Sync command:** `cma` (alias for `chezmoi apply`)
4. **Never edit:** `~/.pi/agent/` directly - changes will be lost

**Two Pi configurations in this repo:**
- **Global:** `pi-agent-base/` → used across ALL Pi sessions on ALL machines
- **Project-local:** `.pi/` → only active when working in this dotfiles repo

**Excluded from sync (preserved in ~/.pi/agent/):**
- `auth.json` - API keys (runtime)
- `sessions/` - Conversation history (runtime)
- `bin/` - Compiled binaries (runtime)
- `git/` - Git packages like the vim-compatible powerline fork (runtime)

**Workflow table added:**
| What | Edit Location | Synced To | Test Command |
|------|--------------|-----------|--------------|
| Pi settings | `pi-agent-base/settings.json` | `~/.pi/agent/settings.json` | `cma && pi` |
| Pi extensions | `pi-agent-base/extensions/` | `~/.pi/agent/extensions/` | `cma && pi` |
| Ansible playbook | `dot_bootstrap/macos.yml` | `~/.bootstrap/macos.yml` | `cma && apConfig` |

This ensures agents always make changes in the right place for persistence across machines.

**See:** `pi-agent-base/WORKFLOW.md` for complete workflow documentation.
