# IdeaLog — 2026-05-10

## Tailscale macOS Variant — Decision Trail

Documenting both switches in one place since the original switch was never
written down (only a vague commit message: `cb0fc19 chore: fix tailscale install
to allow tailscale additional features`).

### Variant landscape (per https://tailscale.com/kb/1065/macos-variants)

| Variant | Package id | SSH server | MagicDNS | GUI |
|---|---|---|---|---|
| App Store (sandboxed) | `io.tailscale.ipn.macos` | ❌ blocked by sandbox | ✅ via NE | ✅ |
| **Standalone "macsys"** | `io.tailscale.ipn.macsys` | ✅ | ✅ via NE | ✅ |
| brew formula `tailscale` | (no .pkg, launchd daemon only) | ✅ | ❌ broken on macOS | ❌ |

### Switch #1 — App Store cask `tailscale` → brew formula `tailscale`

- **When:** commit `cb0fc19` (2026-05-10 — earlier today, retroactive note)
- **Why:** Sandboxed App Store build refused `tailscale set --ssh`:
  > "The Tailscale SSH server does not run in sandboxed Tailscale GUI builds."
  Wanted Tailscale SSH so I could SSH into machines from anywhere without
  managing OpenSSH config / keys.
- **Other "additional features" referenced in the commit:** exit nodes, subnet
  routes — both also worked on the formula.
- **Hidden cost (discovered today):** the brew formula on macOS does NOT
  configure MagicDNS correctly. It writes `/etc/resolver/search.tailscale`
  (search-suffix only) but never the per-domain nameserver file
  (`/etc/resolver/<tailnet>.ts.net` with `nameserver 100.100.100.100`).
  Result: `getaddrinfo(unraid.tail68d30.ts.net)` returns NXDOMAIN even though
  `tailscale status` works fine (status uses the local API socket, not DNS).

### Symptom that triggered Switch #2

`cma` started failing:

```
Unable to fetch ServerConfig from https://unraid.tail68d30.ts.net/vault/api
FetchError: ... getaddrinfo ENOTFOUND unraid.tail68d30.ts.net
```

Verified:

```bash
$ host unraid.tail68d30.ts.net
Host unraid.tail68d30.ts.net not found: 3(NXDOMAIN)
$ ls /etc/resolver/
search.tailscale                       # only file — no per-domain resolver
$ tailscale status                     # works fine, proving daemon is healthy
$ tailscale debug prefs | grep CorpDNS
"CorpDNS": true,                       # daemon thinks DNS is on; macOS disagrees
```

This is a macOS-specific limitation of the formula — the daemon needs the
NetworkExtension entitlement that only the signed .app bundles ship with to
inject per-domain DNS settings system-wide.

### Switch #2 — brew formula → `tailscale-app` cask (macsys standalone)

- **When:** today, 2026-05-10
- **Why:** Restore MagicDNS without losing the features Switch #1 was about.
- **Feature parity (verified against KB 1065):** standalone is strictly a
  superset of the formula on macOS — SSH server ✅, exit nodes ✅, subnet
  routes ✅, Funnel ✅, Taildrop ✅, **plus** working MagicDNS via NE, plus
  optional menu-bar GUI. CLI binary (`tailscale`) is identical.
- **Migration handled in:** `run_once_before_015_install_tailscale.sh.tmpl`
  - Stops `homebrew.mxcl.tailscale` LaunchDaemon
  - Kills lingering `tailscaled`
  - `brew uninstall tailscale` (formula)
  - Cleans `/opt/homebrew/var/lib/tailscale` and `/etc/resolver/*.tailscale`
  - `brew install --cask tailscale-app`
  - Launches the app once so the System Extension approval prompt fires
  - Prints next-step verification commands (`host`, `ls /etc/resolver`,
    `tailscale status | grep ssh`)
- **One-time user friction:** must approve the System Extension once in
  System Settings → Privacy & Security after first launch. After that the
  bootstrap is fully reproducible end-to-end.
- **Re-auth required:** formula state was at `/opt/homebrew/var/lib/tailscale`;
  macsys uses `/Library/Tailscale/`. So `sudo tailscale up --ssh` once.

### Rule for future variant changes

If switching tailscale variants again, *always*:

1. Stop the running daemon before installing the replacement (two `tailscaled`s
   fighting over `utun` interfaces is the cause of half the weird states).
2. Check `pkgutil --pkg-info io.tailscale.ipn.{macos,macsys}` to disambiguate
   App Store vs standalone — `/Applications/Tailscale.app` looks identical
   on the surface but the receipt id is different.
3. Verify MagicDNS with `host <hostname>.<tailnet>.ts.net` and
   `ls /etc/resolver/` — `tailscale status` working is NOT proof DNS works.

## Brew python@3.11 bottle broken on macOS 26 → blocked oMLX install

Surfaced while integrating oMLX (`jundot/omlx/omlx`) into `macos.yml`. After
converting the ansible task to shell-with-logging (so brew's stderr wasn't
truncated), the real error came out:

```
python3.11 -m venv … → ensurepip CalledProcessError
ImportError: dlopen(…/pyexpat.cpython-311-darwin.so):
  Symbol not found: _XML_SetAllocTrackerActivationThreshold
  Expected in: /usr/lib/libexpat.1.dylib
```

macOS 26.2's libexpat reports `XML_ExpatVersion() = expat_2.7.1` but Apple
stripped the alloc-tracker symbols. Brew's bottle was compiled where upstream
expat 2.7.1 (with those symbols) was present, so `pyexpat.so` has an
unsatisfiable undefined reference on this Mac. Affects anything bootstrapping
a venv against `/opt/homebrew/opt/python@3.11`, not just omlx.

Initial fix attempt: `brew reinstall --build-from-source python@3.11`. Did
NOT work — the macOS 26.4 SDK header `expat.h` declares the 2.7.2 symbols
(`/* Added in Expat 2.7.2. */`), so even a from-source pyexpat references
them. Forcing `HOMEBREW_SDKROOT=...MacOSX15.4.sdk` was ignored by brew's own
SDK locator. Would have needed to physically rename the macOS 26 SDKs under
a trap to make brew fall back to 15.4 — brittle and undoes itself on next
brew operation. Abandoned that path.

Decision: drop the brew formula approach for oMLX entirely. Switched the
ansible task to download and install the macOS .dmg app from GitHub releases
(release `v0.3.8`, asset auto-selected by macOS major version: 15-sequoia vs
26-tahoe). The app bundles its own Python + expat, sidestepping the bug.
Only thing we lose is the `omlx` shell command — not needed for the
serve-models-to-tailnet use case, since the menubar + admin dashboard manage
the server lifecycle. Documented in `KNOWN_ISSUES.md`.

Lessons:

1. The ansible homebrew module's stderr truncation cost us a full debug cycle.
   The replacement shell-with-logging pattern surfaced the real error in one
   re-run. Reuse that pattern for any new brittle install task.
2. Don't bake the python rebuild into the playbook — transient bottle bug.
   A pre-flight `python3.11 -m venv /tmp/probe` check that fails fast with a
   pointer to KNOWN_ISSUES is the right shape if this recurs.
3. **Prefer self-contained app bundles over brew formulas for apps that ship
   their own runtime.** brew formulas inherit the toolchain's bugs (and
   Apple's SDK-vs-runtime skew); .dmg apps bundle theirs. The CLI-only
   advantage of a formula is only worth the toolchain coupling if you
   actually script the CLI. For server-style apps you manage via UI / API,
   the .dmg is strictly more robust.
4. Sequence of yak-shaves today: sketchybar/aerospace accessibility →
   tailscale variant misrecommendation → CLT 16 → 26.4 upgrade → brew python
   bottle bug → SDK skew → abandoned formula → .dmg install. Each unrelated
   but all surfaced through `apConfig`. The harness now logs failure tails
   inline, detects stale CLT before install, and prefers .dmg apps for the
   server case, so this kind of debug session should be shorter next time.

## Accessibility code-signing persistence

Side decision made the same session (see commit log / dot_local/bin/
`resign-accessibility-binaries`):

- Root cause of repeated Accessibility re-prompts for sketchybar/aerospace:
  ad-hoc signatures with per-build random identifiers → cdhash changes every
  `brew upgrade` → TCC orphans the previous grant.
- Fix: self-signed `zaye-codesign` cert in login keychain (one-time, manual via
  Keychain Access — Apple's `security` CLI cannot mint code-signing certs).
  Stable signing identity + stable identifier (`net.zaye.{sketchybar,aerospace}`)
  → designated requirement is constant across upgrades → TCC grant persists.
- LaunchAgent `net.zaye.resign-accessibility` watches `/opt/homebrew/bin/{sketchybar,aerospace}`
  and the underlying Cellar/Caskroom dirs; re-signs automatically after any
  `brew upgrade`. `apConfig` also re-signs idempotently as a safety net.
- Ghostty unaffected — it's Developer-ID signed by Mitchell Hashimoto
  (team `24VZTF6M5V`), so its TCC entry persists across upgrades natively.
